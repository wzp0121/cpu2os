在 p0 編譯器中，函數呼叫（Function Call）的運作機制是模擬真實 CPU 的**活動記錄（Active Record）或稱為堆疊幀（Stack Frame）**的概念。

它主要由三個部分協同運作：中間碼生成、參數傳遞、以及 VM 的堆疊切換。以下是詳細的運作原理：

1. 語法解析階段：生成四元組 (Parser & Quadruples)
當編譯器看到 func_name(a, b) 時，它會依序產生幾種指令：

PARAM：每解析出一個實參（Argument），就生成一個 PARAM 指令。這代表「把這個值準備好，待會要傳給函數」。

CALL：解析完所有參數後，生成 CALL 指令，格式通常是 CALL 函數名 參數數量 結果存放變數。

RET_VAL：在函數內部，當看到 return x 時，生成 RET_VAL 指令，將回傳值送回呼叫者。

2. 參數傳遞機制 (Parameter Passing)
在你的程式碼中，參數傳遞是透過一個臨時的參數堆疊 (param_stack) 來完成的：

執行 PARAM 時，VM 把變數的值 push 進 param_stack。

執行 CALL 時，VM 會根據指令中的「參數數量」，從 param_stack 中取出對應的值，存入新函數的「進入參數列表（incoming_args）」。

3. 虛擬機 (VM) 的堆疊幀切換 (Stack Frame Switch)
這是最核心的部分。VM 維護了一個 Frame stack[1000]，每個 Frame 代表一個函數的執行環境（包含它自己的局部變數）。

當執行 CALL 時：
保存現場：記錄目前的 pc + 1 作為回傳位址（ret_pc），並記住要把回傳值存到哪個變數（ret_var）。

切換環境：將堆疊指標 sp 加 1（sp++）。這代表現在進入了一個全新的局部變數空間，函數 A 的變數跟函數 B 的變數完全隔離。

跳轉：將 pc 指向函數定義的起始位置（FUNC_BEG 之後）。

當執行 FORMAL 時：
在進入函數後，前幾條指令通常是 FORMAL。它的作用是從 incoming_args 中把值拿出來，綁定到函數定義的參數名稱上（例如 x, y）。

當執行 RET_VAL 時：
取值：計算出要回傳的數值。

還原環境：將 sp 減 1（sp--），回到呼叫者的環境。

寫回結果：把剛剛計算的回傳值，存入呼叫者環境中指定的 ret_var。

跳回：將 pc 設定為當初保存的 ret_pc，繼續執行後續指令。

4. 運作流程圖解 (以 add(1, 2) 為例)
Caller 視角：

PARAM 1 -> 存入參數堆疊。

PARAM 2 -> 存入參數堆疊。

CALL add, 2, t1 -> 保存現在位置，sp 變 1，跳到 add。

Callee (add) 視角：

FORMAL x -> x = 1 (從參數列表拿)。

FORMAL y -> y = 2 (從參數列表拿)。

ADD x, y, t2 -> t2 = 3。

RET_VAL t2 -> 回傳 3，sp 變 0，跳回 Caller。

Caller 續行：

接收到回傳值 3 並存入 t1。

總結
這種設計的優點是支援遞迴。因為每次 CALL 都會開一個新的 Frame (sp++)，所以函數自己呼叫自己時，
每一層的局部變數都會被獨立儲存在堆疊中，不會互相干擾。
